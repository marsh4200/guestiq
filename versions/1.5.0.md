## v1.5.0 — Admin and staff accounts

GuestIQ now has real accounts instead of one shared password. Sign-in takes a
**username and password**.

### Roles
- **Administrator** — everything, including staff accounts and passwords.
- **Staff** — reception work only. By default a staff member can:
  - see the check-ins dashboard, stats and overdue alerts
  - check guests in (pending arrivals and manual check-in)
  - check guests out and extend a stay
  - add and edit customer records

  Everything else is hidden and blocked: rooms, QR codes, settings, automation,
  updates, deleting anything, and **any password change**.

### Granting extras
Admins open the new **Staff** tab and tick individual permissions per person:
`Add & edit rooms`, `Delete rooms`, `Delete guest records`,
`View & print QR codes`, `Change hotel settings`, `Automation & smart control`,
`Check & apply updates`. Tabs and buttons a person isn't allowed to use don't
render at all, and the API enforces the same rules independently — a blocked
call returns `403` even if someone crafts the request by hand.

### Passwords
- Staff cannot change passwords, including their own. An administrator sets
  them from Staff → **Password**.
- Resetting someone's password (or disabling their account) signs them out
  immediately.
- Settings → *Change my password* is visible to administrators only.

### Staff tab
List of every account with role, granted permissions and last sign-in. Add,
edit, disable, set password, delete. Guard rails: you can't delete or disable
your own account, and the last remaining administrator can't be demoted,
disabled or deleted.

### Also
- New **+ Add customer** button on the Guests tab — staff can add someone to
  the address book without checking them in (`POST /api/guests`).
- `GET /api/settings` no longer returns the automation hub URL or access token
  to accounts without the automation permission.

### Upgrading
On first start the existing admin password becomes an account called **admin**
— sign in with username `admin` and your current password. All existing
sessions are dropped, so everyone signs in once more after the update.

### API
- `POST /api/login` now takes `{username, password}` (username defaults to
  `admin` if omitted)
- `GET /api/me` — current user, permissions and the permission catalogue
- `GET|POST /api/users`, `PUT|DELETE /api/users/{id}`,
  `POST /api/users/{id}/password` — administrator only
- `POST /api/guests` — create a customer record
