## v1.4.3 — Full-house guard & the "invalid code" menu bug

### No rooms available
- Trying to check anyone in — pending arrival or manual — when every room is
  occupied now shows a proper **No rooms available** dialog listing which rooms
  are in use and who is in them, with a shortcut to the Rooms tab. Previously
  the form opened with an empty room dropdown and only complained "Pick a room"
  on submit.
- If no rooms have been set up at all, the dialog says so instead.
- Room state is re-fetched the moment either dialog opens, so the check can't
  act on a stale list.
- **The API now refuses double-bookings too.** `POST /api/stays` and
  `/api/stays/{id}/assign` return `409` — *"Room 4 · Kudu is already occupied by
  Piet van Wyk — check them out first"* — if the room already holds a checked-in
  guest. Until now nothing stopped two guests being put in the same room.
- New `GET /api/rooms/availability` returning `total` / `occupied` /
  `available` / `full`.

### Fix: "Room not found — this code is invalid" after scanning a room QR
- Cause: an online menu URL saved without `https://` (e.g. `lodge.co.za/menu`).
  The browser treats a scheme-less link as a path relative to the page it sits
  on, so **View Menu** on `/room/3-kudu` navigated to `/room/lodge.co.za/menu`
  — a room code that doesn't exist — and the guest got the invalid-code page.
- URLs saved in Settings are now normalised (`https://` added when missing), the
  room page normalises defensively before rendering the link, and any URL
  already stored without a scheme is repaired automatically on first start.
- Menu URL field in Settings gained a note explaining this.
